# MEMORY.md

本文件是项目长期记忆库，用来保存 AI 无法仅靠代码直接推断出来的业务背景、口径、历史决策和关键数据知识。

不要在这里记录临时任务；临时任务写入 `TASKS.md`。不要在这里记录开发规则；开发规则写入 `AGENTS.md`。

## 项目背景

本项目服务于榴莲运输业务，重点关注从下单、装柜、国外在途、到岸、国内短驳、签收到终端的全过程。业务人员希望通过看板快速判断柜子当前卡在哪个环节、是否温度异常、是否需要催办。

示例：

- 业务人员打开基础看板，要能看到泰国、越南等来源国的总柜数、到岸、签收、在途和待发情况。
- 业务人员打开货柜流转看板，要能看懂总柜数如何拆分为已发货、未发货、国外在途、已到岸、在岸、国内短驳、国内在途和已签收。

## 业务背景

榴莲业务存在鲜果和冻果两类核心品类。订单先在订单主表产生下单柜数，实际发货后才会在分柜明细表中形成柜号。一个订单可以有多个柜号，同一个柜号在特定场景下可能因为不同发货时间代表多次装柜。

示例：

- A 订单下有柜号 1、柜号 2、柜号 2、柜号 3、柜号 3。
- 柜号 2 的发货日期分别是 1 号和 9 号，代表同一柜号装了两次，应按两柜货统计。
- 柜号 3 的发货日期都是 5 号，代表同一柜货在国内拆分配送，应按一柜货统计。

## 核心业务规则

### 货柜流转关系

```json
{
  "总柜数": {
    "已发货": {
      "国外在途": {},
      "已到岸": {
        "国内短驳": {
          "国内在途": {},
          "已签收": {}
        },
        "在岸": {}
      }
    },
    "未发货": {}
  }
}
```

解释：

- `总柜数`：订单主表中所有订单的下单柜数，等于鲜果柜数加冻果柜数。
- `已发货`：分柜明细中有柜号，按同一订单下不同发货时间的去重柜号条数统计。
- `未发货`：总柜数减已发货柜数。
- `已到岸`：分柜明细中到岸日期或到岸时间非空。
- `国外在途`：已发货但未到岸。
- `国内短驳`：已到岸且国内发货时间非空。
- `在岸`：已到岸但无国内发货时间、无签收时间。
- `国内在途`：有国内发货时间但无签收日期。
- `已签收`：签收日期非空。

示例：

- 某柜有柜号和发货日期，没有到岸日期：属于 `国外在途`。
- 某柜有到岸日期，没有国内发货时间和签收日期：属于 `在岸`。
- 某柜有国内发货时间，没有签收日期：属于 `国内在途`。

### 基础看板统计口径

基础看板的统计逻辑位于 `lib/wecom-aggregate.js`，详细说明见 `docs/statistics-logic.md`。

默认原则：

- 总箱量来自 `订单主表` 的鲜果柜数和冻果柜数。
- 到岸、签收、在途等来自 `分柜明细表`。
- 分柜明细统计应按订单、品类、柜号、发货时间去重。
- 交付率不能超过业务合理上限，避免把签收和到岸重复叠加。
- 国家维度的订单数按订单主表唯一订单统计，不按鲜果/冻果拆行相加，避免双品类订单重复计数。

示例：

- 订单主表显示某订单鲜果 10 柜，分柜明细已登记 6 柜，其中 4 柜到岸，则到岸口径下交付率为 40%。
- 如果后台把分子改成签收柜数，则同一订单按签收数量计算。

### 温度业务规则

温度记录来自 `温度记录` 子表。核心关注设定温度、送风温度、回风温度、当前位置、味道、关口和更新时间。

示例：

- 回风温度与设定温度偏差超过阈值时，视为温度异常。
- 人员输入自然语言“回风14度，味道淡香，状态没问题”时，AI 智能解析应提取回风温度、味道、状态等字段，未识别字段留空供人工确认。

### 物流监控口径

物流监控里的“在途批次”与“关口滞留”不能只看温度记录或陆运记录，必须结合分柜明细的当前状态判断。

示例：

- `在途批次`：按分柜明细中当前仍处于国外在途状态的柜数统计。
- `关口滞留`：按分柜明细中已到岸、未国内发货、未签收且超过滞留阈值的柜数统计。
- 温度记录只用于展示在途柜温度与告警，不作为在途/滞留主口径。

## 已确认的设计方案

### 企业微信智能表为主要业务数据源

企业微信智能表是运输、订单、温度、物流数据的主要来源。系统应尽量从智能表读取真实数据，并通过缓存降低穿透查询和看板刷新延迟。

示例：

- 看板每 30 秒刷新时，不应每次用户点击穿透都重新通拉所有数据；优先使用内存快照缓存。
- 穿透弹窗查看订单或柜号时，优先从缓存快照读取。

### 运行时配置优先

可变口径尽量放到后台配置，不硬编码在代码里。字段映射应优先保存字段 ID，其次才使用字段标题兜底。

示例：

- 统计设置允许配置交付率分子、分母、去重方式。
- 流转看板允许配置订单主表、分柜明细表和关键字段。
- 如果字段标题被改名但字段 ID 不变，统计不应失效。

### AI 智能解析必须人工确认

自然语言或截图解析出的信息不能直接写入智能表。系统必须先展示将写入哪张子表、哪条记录、哪些字段、是新增还是修改，再由人员确认。

示例：

- 输入“南部第10柜榴小杰昨天傍晚装柜发出”后，AI 可以建议写入温度记录表并默认新增，但必须让人员确认。
- 修改记录时，系统应帮助候选记录 ID，但人员可以重选。

### 新闻抓取要控制成本

自动新闻可以使用免费源、搜索引擎和秘塔搜索，但秘塔调用需要严格控制次数和成本。没有必要时优先使用低成本来源。

示例：

- 秘塔应有启用开关、每日调用上限和冷却时间。
- 新闻源展示时要标注真实来源，不能把秘塔搜到的链接错误标为搜狗来源。

## 重要技术决策

### 内存快照缓存

`lib/wecom-cache.js` 保存企业微信智能表内存快照，默认 30 秒刷新一次。第一版不落本地文件，刷新成功后整体替换快照；刷新失败时继续使用上一份成功快照。

示例：

- `/api/flow-dashboard` 使用 `wecomCache.getSnapshotOrRefresh(process.env.DOCID)` 获取快照。
- 穿透查询应优先从快照读取，减少实时 API 请求。

### 前端以单文件页面为主

现有看板和后台大多是单文件 HTML，CSS 和 JS 内联。短期内修改应遵循这个模式，避免引入构建系统。

示例：

- 修改货柜流转看板时编辑 `public/index-flow.html`。
- 修改基础看板时编辑 `public/index.html`。
- 修改后台智能表逻辑时注意 `public/admin.html` 与 `public/admin-smartsheet.js` 的全局变量冲突。

### 认证和权限

系统使用 JWT + httpOnly Cookie。所有 `/api/*` 默认需要登录，写操作和管理操作需要管理员或相应权限。

示例：

- 未登录访问页面时，前端应跳转 `/login?redirect=<当前页面>`。
- 智能解析写入、刷新新闻、修改配置等操作不能开放给只读用户。

## 开发过程中积累的重要知识

### 企业微信日期写入

企业微信智能表 DATE_TIME 字段写入时应使用字符串形式的毫秒时间戳，不要用数字类型。

示例：

```js
// 正确
values['更新时间'] = String(Date.now());

// 不推荐：可能接口返回成功但单元格不显示
values['更新时间'] = Date.now();
```

### 引用字段显示值不等于真实业务值

企微引用字段可能返回记录 ID、展示文本或嵌套对象。看板展示订单时不能直接把引用字段的值当订单号，应映射回订单主表真实订单编号。

示例：

- 分柜明细中订单字段显示 `Vtfala`，这可能只是订单主表记录 ID。
- 系统应根据订单主表记录 ID 映射到真实订单编号，如 `KK-20260401`。

### 时间展示

业务页面不应显示裸时间戳。日期应按业务可读格式展示，有小时分钟时展示到分钟。

示例：

- 仅日期：`26-04-07`
- 日期时间：`26-04-07 18:30`

### 新闻抓取诊断

新闻为 0 时需要区分：

- 是否没有发出请求。
- 是否请求失败。
- 是否抓取到结果但被关键词过滤。
- 是否因为 API 限额或冷却策略跳过。

示例：

- 自动新闻刷新后应在诊断信息里显示来源、命中数量、过滤数量、错误信息和最后更新时间。
